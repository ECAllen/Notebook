Ch3 How to Model Services

What makes a good service?
  • loose coupling - change in one service does not require change in another service
  • high cohesion - related behaviour together, unrelated behaviour elsewhere
  • bounded context - models,objects that do not need to be communicated outside as well as models that are shared externally. You communicate with it's explicit boundary using models. Reference Domain-Driven Design.

Ch 4 Integration

Synchromnous vs Asynchronous

Orchestration vs Choreography
  • orchestration - central process directs the app
  • choreography - each part knows it's job and they work out the details

RPC vs REST

RPC - technique of making local call and having it execute a remote service.

RPC downsides
  • networks arent reliable which can be a problem when coding as if remote service is local call
  • brittleness - clients become dependent on implementations on the server, changes on the server require changes on the client ( tight coupling )
